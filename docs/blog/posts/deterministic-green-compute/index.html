TL;DR

I gave CJC-Lang a green-compute layer in three phases. Phase 1 is a Runtime Policy Layer: --profile cool|balanced|max-perf bounds how much CPU a run uses, plus a deterministic energy estimate. Phase 2 is race-to-idle adaptive scheduling — full speed for short bursts, throttle only when load is sustained — which recovered ~1.8× of the throughput the fixed cap cost, while keeping the sustained thermal bound. Phase 3a is fused elementwise kernels that eliminate intermediate tensor allocations. The whole thing rests on one invariant: thread count, pool size, and the adaptive schedule provably never change the output. Same seed, same bits — across every profile, thread count, and both executors. That’s what makes an adaptive thermal knob safe in a language whose entire pitch is reproducibility.

The problem: a laptop that cooks itself

CJC-Lang’s core promise is same seed, same output, always. To get there it already does the unglamorous work: ordered collections, Kahan/binned summation, a seeded RNG, and — critically for this story — no FMA in the SIMD kernels, so a multiply-then-add rounds the same way every time.

But nothing in the runtime bounded how hard it pushed the machine. The parallel matmul and reduction kernels ran on Rayon’s default global pool — every core, all the time. On a laptop, a training loop would peg all 8 cores, spin the fans up, and eventually thermal-throttle: the CPU gets hot enough that the firmware lowers the clock to protect itself, and you end up slower than if you’d never saturated it.

I wanted an explicit, deterministic answer to “how much machine may this run use?” — without the OS power governor’s opacity, and without breaking the reproducibility contract.

The reframe that makes it tractable

Here’s the mental model I kept coming back to:

Energy = power × time. Capping cores cuts power but raises time. On a short burst that’s often a wash — you run cooler but longer. The levers that recover performance and save energy attack the other terms: total work (fewer cycles), memory traffic (the dominant energy term), and when you spend the thermal budget.

Crucially: bursts don’t overheat — only sustained load does. Thermal time constants are seconds to tens of seconds, so a sub-second burst at full tilt never heat-soaks the package. A fixed thread cap pays the throttle tax on every operation, including the short interactive runs that were never going to overheat.

And the linchpin that makes all of this safe: in CJC-Lang, thread count never changes the result. The parallel reductions sum in a fixed chunk order with Kahan/binned accumulators, so the numeric output is identical no matter how many workers are live. That single guarantee is what lets me treat parallelism as a pure heat/speed knob — and even make it adaptive — without ever risking the answer.

Phase 1: a Runtime Policy Layer

The first piece is a RuntimePolicy — the green-compute control surface. It lives in a thread-local in cjc-runtime, mirroring the existing profiler sink.

pub struct RuntimePolicy {
    pub determinism: Determinism,   // Strict
    pub numeric_mode: NumericMode,  // Kahan | Binned | FixedTree
    pub thermal_mode: ThermalMode,  // Cool | Balanced | MaxPerf
    pub max_threads: usize,         // 0 = auto (derive from thermal_mode)
    pub batch_size: usize,
    pub audit_mode: AuditMode,      // Summary | Full | Forensic
    pub adaptive: bool,             // race-to-idle (Phase 2)
}

Three thermal profiles, exposed as CLI flags applied once at startup so they cover run, bench, and every command:

Profile Threads (8-core host) Batch Use
cool ≈ ¼ cores → 2 32 laptop, edge, gentle
balanced (default) ≈ ½ cores → 4 128 normal — leaves headroom
max-perf all → 8 512 benchmarking

The default is balanced, deliberately not “all cores forever.”

A deterministic energy estimate

Most energy tools (CodeCarbon, Scaphandre, RAPL counters) measure power from hardware — which is non-deterministic and machine-specific. I wanted energy to be a number you can assert in a test. So energy_estimate is a pure function of the workload, never of wall-clock time:

joules ≈ flops × 1e-10  +  bytes × 1e-10
         (~100 pJ/flop)     (~100 pJ/byte, DRAM-ish)

The two terms are computed in separate bindings so the compiler can’t fuse them into an FMA — the same no-FMA discipline as the SIMD kernels. Same program, same seed → same FLOP count → same joules, bit-for-bit. On the branch, energy_estimate(1000000, 2000) returns 0.00010020000000000001 — identical under every profile, thread count, and both executors. The absolute number is an order-of-magnitude estimate; the ratio between two runs is the meaningful, reproducible figure.

What the fixed cap costs

Measured on an 8-core machine, release build, cjcl bench (10 runs) on a 1024×1024 matmul:

Profile Threads Mean
--threads 1 1 628 ms
cool 2 468 ms
balanced (default) 4 268 ms
max-perf 8 210 ms

So the cap genuinely throttles (max-perf is ~3× faster than single-threaded) — and the laptop-safe default costs ~27% versus all-out. That cost is real, and Phase 2 is about clawing most of it back.

Phase 2: race-to-idle adaptive scheduling

A fixed cap throttles short and long workloads alike. Race-to-idle fixes that: run at full width during a burst, and throttle to the cap only once load is sustained (a 2-second window; a ½-second idle gap resets it).

The mechanism is the interesting part. Rayon’s global pool can only be configured once, so I don’t shrink it — I leave it at full size so bursts can use every core. Throttling happens per-operation: when a workload has been busy past the window, the kernel’s parallel section is install-ed into a cached, cap-sized pool. Inside that pool rayon::current_num_threads() reports the cap, so the existing band/row/chunk splitters auto-scale — no per-kernel rewrites.

pub fn run_parallel<R: Send>(work: impl FnOnce() -> R + Send) -> R {
    if rayon::current_thread_index().is_some() { return work(); } // nested → inline
    let cap = effective_threads(&get(), detect_cores());
    if cap >= detect_cores() { return work(); }                   // no cap
    let throttle = if get().adaptive { is_sustained_now() } else { true };
    if !throttle { return work(); }                               // burst → full pool
    match capped_pool(cap) { Some(p) => p.install(work), None => work() }
}

The four parallel kernels (large/medium/batch matmul, SIMD elementwise) wrap their bodies in run_parallel.

Did it work? Two regimes, measured

Short workload (one 1024² matmul, finishes inside the burst window), minimum of 9 --time runs:

cool fixed cool adaptive max-perf
time 0.353 s 0.196 s 0.153 s

Adaptive-cool recovered ~1.8× (0.353 → 0.196 s) — landing within fixed startup overhead of max-perf. The short, interactive case now runs at full speed and never trips the cap.

Sustained workload (25× 1024² matmuls, ~runs well past the 2 s window):

cool fixed cool adaptive max-perf
time 16.7 s 10.8 s 4.2 s

This is the part I care about most: adaptive-cool (10.8 s) sits between fixed-cool and max-perf. It is not near max-perf (4.2 s) — meaning it genuinely throttled after the burst. The thermal bound held; the burst just bought back the first couple of seconds. Exactly the goal: fast when it’s safe, bounded when it matters.

Why this is safe

The burst-vs-throttle decision reads the wall clock, so the schedule is non-deterministic run-to-run. But the output isn’t, because the choice only changes how many bands run concurrently — never the per-element math. To prove it rather than assert it: a 256×256 matrix product (a real parallel reduction over varied values) hashed byte-identical across --threads 1, cool, balanced, max-perf, adaptive on/off, and both the AST and MIR executors:

threads=1  (eval)    : 79aaf9931d2b6c2004a20cdf5d5bd434
threads=8  (eval)    : 79aaf9931d2b6c2004a20cdf5d5bd434
cool (adaptive burst): 79aaf9931d2b6c2004a20cdf5d5bd434
cool --no-adaptive   : 79aaf9931d2b6c2004a20cdf5d5bd434
max-perf             : 79aaf9931d2b6c2004a20cdf5d5bd434
MIR cool (adaptive)  : 79aaf9931d2b6c2004a20cdf5d5bd434

For reproducible benchmarking — where you want a fixed schedule — --no-adaptive applies the cap uniformly.

Phase 3a: fused kernels cut memory traffic

The third lever is memory. A plain a*b + c allocates an intermediate: mul → a fresh tensor, then add → another. That’s two allocations and ~5N words of traffic for an N-element result. A fused single-pass kernel does it in one allocation and one pass (~4N words).

I added three, joining the existing broadcast_fma:

Builtin Computes Replaces
fused_axpy(α, x, y) α·x + y scalar-mul + add (BLAS axpy)
fused_mul_sub(a, b, c) a·b − c mul + sub
fused_sub_sq(a, b) (a−b)² sub + mul (MSE/variance/distance)

The subtle bit is why this is free: a*b + c in Rust compiles to a separate multiply then add — it does not contract into a hardware FMA unless you call .mul_add() explicitly. So the fused loop produces bit-identical results to the two-pass version while eliminating the intermediate. I pinned that with property tests asserting fused == unfused byte-for-byte over 256 random cases each.

And this compounds with Phase 2: lighter work finishes inside the burst window, so the cap engages even less. Memory efficiency and the adaptive scheduler reinforce each other instead of trading off.

How unique is this?

Honest answer: the individual pieces all exist somewhere. Thread caps are everywhere (Rayon, OpenMP, GOMAXPROCS). OS governors and RAPL do power capping. Race-to-idle is textbook power management. CodeCarbon and friends estimate energy. Intel MKL has reproducible-numerics modes.

What I haven’t seen packaged together is the combination at the language-runtime level with a determinism guarantee: a thermal/energy policy where the knobs are provably result-invariant, paired with an adaptive scheduler and a reproducible energy estimate. In most stacks, changing thread count can change floating-point results; here it provably can’t, which is the whole reason an adaptive thermal scheduler is even safe to ship. It’s most at home in the niche CJC-Lang targets — reproducible scientific, regulated, and edge computing — not hyperscale throughput.

What I measured vs. what I estimated

In keeping with the project’s ethos, the honest accounting:

  • Measured: all the wall-clock numbers (release cjcl bench / --time), the determinism hashes, the test counts. These are real and on the branch.
  • A proxy, not a measurement: I measured wall-clock and active-core counts, not watts or junction temperature. Fewer active cores means less package power, but I don’t have hardware sensors in the loop — treat the thermal claims as a well-founded proxy, not a power reading.
  • Analytical, not benchmarked: the fused-kernel traffic reduction (~5N → ~4N words, two allocations → one) is from counting, not a wall-clock microbenchmark. What I did verify is that the output is bit-identical to the unfused path.
  • Deferred: the data-DSL group-by/join memory work (Phase 3b) is not in this post. The obvious approach turned out to change output order (the streaming path sorts keys; the materializing path preserves first-occurrence order), which would break determinism — so I stopped rather than ship a correctness regression. It needs an order-preserving design, which is its own phase.

Verification

Every claim above is backed by tests on the branch: 80 new tests for this work (20 unit + 43 integration for the policy/scheduling layer; 17 for the fused kernels), the dual-executor parity gate at 10/10, and the full workspace library suite at 2653 passing, 0 failures. No determinism gate moved.

The throughline: you don’t have to choose between fast, reproducible, and cool. You can have a thermal bound that adapts to your workload, energy you can reason about, and lower memory traffic — and still get the same bits out every single time. In a deterministic language, that last clause isn’t a nice-to-have. It’s the whole point.
