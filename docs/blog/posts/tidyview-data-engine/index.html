TidyView: A Database Engine in Disguise

How CJC-Lang handles data manipulation — and why I couldn’t just use Pandas.

TL;DR

TidyView is CJC-Lang’s data manipulation system. It looks like R’s dplyr on the surface, but underneath it’s a database engine: columnar storage, lazy evaluation, bitmask filtering, predicate pushdown, and zone maps. The result is a data pipeline where every intermediate state is inspectable, the original data is never mutated, and the same input always produces the same output — down to the last bit.

The Problem

When I started building CJC-Lang, the core promise was simple: same seed, same output, always. Every part of the language enforces this — ordered collections, Kahan summation, seeded RNG, no FMA in SIMD kernels.

But then I needed to handle data. Filter a table. Group rows. Compute aggregates. Join two datasets.

The obvious move was to reach for an existing DataFrame library. But when I looked at what’s available, I kept running into the same problems:

Hash-based grouping. Most DataFrame libraries — including Pandas and Polars — use hash maps to group rows. Hash maps are fast, but their iteration order may depend on internal hash seeds or execution strategy. Depending on the configuration, grouping the same data twice can produce rows in a different order. That makes bit-identical reproducibility difficult to guarantee.

Floating-point reduction order. Call sum() on a million floats in Pandas. The result depends on the order of accumulation. Add the numbers in a different order and you get different rounding errors. Small differences, but they compound — and they mean “same data, different result” depending on implementation details. (Polars and other libraries may use more sophisticated reduction strategies, but the result can still vary across execution modes or platforms.)

Parallel scheduling. Libraries that use thread pools for parallel execution (Polars uses Rayon, for example) gain significant throughput. But thread scheduling is inherently non-deterministic, which means parallel aggregation can introduce ordering variation between runs — even with identical input. Whether this matters depends on the workload, but for bit-identical reproducibility it’s a concern.

Aliasing mutations. Pandas has an entire warning category (SettingWithCopyWarning) dedicated to the fact that modifying a filtered subset might or might not modify the original DataFrame. Whether it does depends on internal memory layout decisions that are invisible to the user.

None of these are bugs. They’re design trade-offs made by libraries that prioritize speed over reproducibility. But they break CJC-Lang’s core contract. So I built something else.

What TidyView Is

TidyView is a three-layer data manipulation system:

Layer Type What it does
Immutable view TidyView Zero-copy lens: bitmask + projection over shared data
Materialized frame TidyFrame Copy-on-write buffer for mutations
Grouped view GroupedTidyView Group index for aggregation

The API surface is modeled on R’s dplyr — filter, select, mutate, group_by, summarise, arrange, join. If you’ve used the tidyverse, you already know the vocabulary.

But the implementation borrows from a different tradition entirely: analytical databases.

The Database Connection

I didn’t set out to build a database engine. But the constraints I was working under — determinism, auditability, memory safety — kept pushing me toward the same solutions that database engineers arrived at decades ago.

Columnar Storage

Data is stored column-major:

// Internally:
columns: [
    ("name",    Str(["Alice", "Bob", "Carol", "Dave"])),
    ("age",     Int([30, 25, 35, 28])),
    ("revenue", Float([150.0, 200.0, 175.0, 225.0])),
]

Each column is a contiguous typed buffer. When you compute sum(revenue), the runtime scans a single Vec<f64> — no pointer chasing, no skipping over unrelated fields. This is the same storage model used by Apache Arrow, DuckDB, and Parquet.

The alternative — row-major storage — would scatter each revenue value across a row struct alongside name and age data. For a table with 100 columns, computing a single-column aggregate would touch 100x more memory than necessary.

Bitmask Filtering

This is the design choice that makes everything else work. When you filter a TidyView, no data is copied. Instead, a bitmask records which rows passed the predicate:

// Original data: 4 rows
base: ["Alice", "Bob", "Carol", "Dave"]  // ages: 30, 25, 35, 28

// After filter(age > 28):
mask: [1, 0, 1, 1]  // Alice (30), Carol (35), Dave (28 fails, hypothetical: > 28)

In practice the mask is packed into 64-bit words — one bit per row. A million-row filter costs 125 KB of bitmask memory. The original data stays exactly where it was, untouched.

Chain three filters together? The bitmasks are ANDed. Still no data copied. Still 125 KB.

let sales: Any = read_csv("sales.csv");
let df: Any = sales
    |> filter(col("region") == "West")
    |> filter(col("revenue") > 1000.0)
    |> filter(col("year") == 2025);

Each filter call returns a new TidyView with a new bitmask. All three views share the same underlying data. The CSV is loaded once. This is the same technique used by bitmap indexes in Oracle and selection vectors in DuckDB.

Predicate Pushdown

TidyView has a lazy evaluation layer where operations build a logical plan tree before executing. The optimizer rewrites this tree using three classic database passes:

Pass 1 — Filter merging. Consecutive filters are combined into a single filter with an AND predicate.

Pass 2 — Predicate pushdown. Filters are pushed as close to the data source as possible.

Pass 3 — Redundant select elimination. Select nodes that project all columns are removed.

Here’s why this matters. Suppose you write:

let result: Any = df
    |> mutate("bonus", col("salary") * 0.1)
    |> filter(col("department") == "Engineering");

Without optimization, this computes bonus for every row, then filters. With predicate pushdown, the optimizer recognizes that department == "Engineering" doesn’t depend on the bonus column, so it pushes the filter below the mutate. Now filtering happens first, and bonus is only computed for engineering rows.

The pushdown rules follow standard relational algebra:

  • Push past select — always safe
  • Push past arrange — filtering doesn’t affect sort order
  • Push past mutate — only if the predicate doesn’t reference a mutated column
  • Push into join branches — if the predicate references only one side
  • Never past group_by + summarise — aggregation changes row identity
  • Never past distinct — uniqueness changes row identity

These are the same rules taught in database systems textbooks. They apply here because TidyView’s operation set is relational algebra — just with tidyverse names.

Zone Maps

Each column tracks lightweight statistics:

// ColumnStats for the "revenue" column:
min: 150.0
max: 225.0
sorted_ascending: false
n_distinct: 4

When you write filter(col("revenue") > 500.0), the zone map says: max is 225, so no row can satisfy > 500. The entire filter can be skipped.

When a column is sorted, the system uses binary search instead of a linear scan — O(log N) instead of O(N).

These are the same optimizations that Vertica, Redshift, and Parquet use at the storage layer. In TidyView they operate at the column level because the entire “database” fits in memory.

What This Prevents

The architectural choices above aren’t just about performance. They prevent specific classes of bugs that plague traditional DataFrame workflows.

The Aliasing Problem

In Pandas:

df = pd.DataFrame({"x": [1, 2, 3], "y": [4, 5, 6]})
subset = df[df.x > 1]
subset["z"] = 99      # Does this modify df? Maybe.

Whether subset["z"] = 99 modifies the original df depends on whether Pandas chose to return a view or a copy when you wrote df[df.x > 1]. This is the SettingWithCopyWarning — one of the most common sources of confusion in data science code.

In TidyView, this problem doesn’t exist. The base data lives behind an Rc (reference-counted pointer) and is never modified. When you need to mutate, TidyFrame checks the reference count:

  • If only one reference exists, mutate in place
  • If multiple references exist, deep-clone first, then mutate the clone

The check is deterministic — it’s a simple integer comparison, not a heuristic. The other views are never affected.

The Order-Dependent Mutation Problem

In Pandas:

df = df.assign(a = df.x + 1, b = df.a * 2)

Does df.a in the second expression see the newly computed a or the original a? In Pandas, it sees the new one. The result depends on the order of the assignments. Swap a and b and you get different data.

In TidyView, mutate uses snapshot semantics: all column references are validated against the column list frozen before any assignments execute. Every expression sees the data as it was when the mutate call started:

let result: Any = df
    |> mutate("a", col("x") + 1)
    |> mutate("b", col("a") * 2);
// col("a") always reads the ORIGINAL "a" column
// If "a" didn't exist before, this is an error — not a silent dependency

This is the same principle as SQL’s UPDATE SET a = b, b = a — the SET clause reads from a snapshot, so this correctly swaps the values rather than setting both to b.

The Hidden Null Problem

Pandas DataFrames can have NaN or None in any column at any time. You discover this at runtime, usually when an aggregation silently skips missing values and gives you a number that’s subtly wrong.

In TidyView, the base Column type has no null representation:

// Column — always fully populated:
Int(Vec<i64>)
Float(Vec<f64>)
Str(Vec<String>)
Bool(Vec<bool>)

There’s no Option wrapper. No sentinel values. If you have a TidyView, its data is guaranteed null-free by type.

Nulls exist, but they’re quarantined in a separate type called NullCol. You only encounter NullCol through operations that can introduce nulls — left joins (unmatched rows), pivot_wider (missing combinations), full joins. These operations return a NullableFrame instead of a TidyFrame, and the type boundary makes the null-possibility explicit.

The question “does my data have nulls?” is answered by the type, not by scanning every cell.

The Floating-Point Drift Problem

# Pandas
pd.Series([1e16, 1.0, -1e16]).sum()  # → 0.0 (wrong)

The mathematically correct answer is 1.0. Pandas gives 0.0 because it uses naive left-to-right summation: 1e16 + 1.0 rounds to 1e16 (the 1.0 is lost to floating-point precision), then 1e16 - 1e16 = 0.0.

TidyView uses Kahan compensated summation for every floating-point reduction — sum, mean, variance, sd, all of them. Kahan summation tracks a correction term that captures the rounding error:

// Kahan summation:
//   sum = 0, c = 0
//   y = 1e16 - 0 = 1e16;    t = 0 + 1e16 = 1e16;    c = (1e16 - 1e16) - 1e16 = 0
//   y = 1.0 - 0 = 1.0;      t = 1e16 + 1.0 = 1e16;   c = (1e16 - 1e16) - 1.0 = -1.0
//   y = -1e16 - (-1.0);      ...compensates the lost 1.0
//   Result: 1.0 (correct)

Same input, same result, every time. Not approximately correct — exactly correct within the limits of double precision.

How to Audit a TidyView Pipeline

This is where the architecture pays off in practice. When something goes wrong in a data pipeline, the question is always: where did the data go wrong?

In TidyView, every intermediate step is inspectable because the original data is never modified. Every filter produces a bitmask you can inspect — count the set bits, iterate them to see which rows survived, AND two masks to see which rows passed both filters. Every group_by produces a group index with key values and row indices. And the original data stays intact behind a shared reference, so you can always go back and check it.

This is fundamentally different from a system where df = df[df.region == "West"] discards the filtered-out rows. In that system, if the filter was wrong, the evidence is gone. In TidyView, the evidence is the bitmask, and the original data is still there to check against. (A full worked example with concrete row indices is in the Auditing a Pipeline section below.)

Error Messages That Point to the Problem

When something goes wrong, TidyView’s error types name the exact issue:

ColumnNotFound("revnue")       // Typo — you meant "revenue"
DuplicateColumn("total")       // Two columns with the same name in select
PredicateNotBool { got: "Int" } // filter predicate returned Int, not Bool
TypeMismatch { expected: "Float", got: "Str" }  // arithmetic on a string column
LengthMismatch { expected: 1000, got: 500 }     // broadcast size mismatch
EmptyGroup                     // first() or last() called on a group with zero rows
CapacityExceeded { limit: 65535, got: 70000 }   // too many distinct levels for a categorical column

Every error variant names the specific column, type, or value that caused the problem. You know what went wrong and where, not just that something failed.

EmptyGroup catches a subtle bug: if an aggressive filter removes all rows from a group and you then call first() or last() in your summarise, you get a clear error instead of a silent null or a crash. CapacityExceeded guards the categorical column type — CJC-Lang’s factor columns use u16 codes (max 65,535 distinct levels) for cache efficiency, and this error tells you exactly when your data exceeds that budget.

Why Not Just Use Polars?

Polars is a good library. It’s fast, it has a lazy API, and it uses Arrow-backed columnar storage. If I were building a Python application, I’d probably use it.

But Polars makes different trade-offs than TidyView:

Polars TidyView
Grouping Hash-based (fast; order may vary by execution mode) BTreeMap (deterministic first-occurrence order)
Summation Optimized reduction (fast; may vary across platforms) Kahan compensated (deterministic, bit-identical)
Parallelism Rayon thread pool (high throughput) Single-threaded (deterministic, no scheduling variance)
Null model Nullable everywhere (Arrow validity bitmaps) Null-free by default, NullCol only where needed
Sampling System RNG Seeded LCG-based Fisher-Yates

None of Polars’ trade-offs are wrong — and Polars is significantly faster for most workloads. It’s a production-grade library optimized for throughput. TidyView is optimizing for a different property: reproducibility — the guarantee that the same pipeline on the same data always produces the same result, bit for bit, on any machine. TidyView will be slower than Polars on almost any benchmark. That’s the expected cost of determinism.

When your work requires an audit trail — when a regulator, a reviewer, or future-you needs to verify that a result is correct — that guarantee can matter more than raw speed.

TidyView in Practice

The architecture above is interesting in the abstract, but what does it actually look like when you write code? Here’s a walkthrough of real CJC-Lang programs using TidyView.

Loading and Filtering

let csv = "name,age,city\nAlice,30,NYC\nBob,25,LA\nCharlie,35,NYC\nDiana,28,LA\n";
let df = Csv.parse(csv);
let view = df.view();

// filter: age > 28 (keeps Alice=30, Charlie=35)
let older = view.filter(dexpr_binop(">", col("age"), 28));
print(older.nrows());   // → 2

Csv.parse() returns a DataFrame. .view() wraps it in a TidyView — a zero-copy lens over the data. The filter call doesn’t copy any rows. It creates a new bitmask: [1, 0, 1, 0]. The original four rows are still sitting in the same memory, untouched.

Chaining Operations

let csv = "dept,name,salary\nEng,Alice,100\nEng,Bob,120\nSales,Carol,80\nSales,Dave,90\nEng,Eve,110\n";
let df = Csv.parse(csv);
let view = df.view();

let result = view
    .filter(dexpr_binop(">", col("salary"), 85))
    .group_by(["dept"])
    .summarise("n", tidy_count(), "avg", tidy_mean("salary"));
print(result.nrows());   // → 2 (Eng and Sales groups)

This is a three-step pipeline: filter rows where salary exceeds 85, group surviving rows by department, then compute count and mean per group. The filter produces a bitmask; the group-by reads through it to build its index; the summarise collapses each group into one row.

Adding Computed Columns

let mutated = view.mutate("bonus", dexpr_binop("*", col("salary"), 10));
print(mutated.ncols());   // → 4 (dept, name, salary, bonus)

mutate returns a TidyFrame — a materialized copy with the new column appended. The original view still has three columns. This is the copy-on-write boundary: reads are free, writes trigger a copy.

Sorting and Slicing

let csv = "name,score\nAlice,85\nBob,92\nCharlie,78\nDiana,95\n";
let df = Csv.parse(csv);
let sorted = df.view().arrange([desc("score")]);
let top2 = sorted.slice_head(2);
print(top2.nrows());   // → 2 (Diana=95, Bob=92)

arrange materializes the data in sorted order (sorting can’t be lazy — it needs to compare every element). slice_head takes the first N rows from the result.

Joins

let left = Csv.parse("id,name\n10,Alice\n20,Bob\n30,Charlie\n").view();
let right = Csv.parse("id,dept\n10,Eng\n30,Sales\n40,HR\n").view();

let joined = left.inner_join(right, "id", "id");
print(joined.nrows());   // → 2 (id=10 and id=30 match)

Inner join keeps only rows where the key exists in both tables. Alice (id=10) matches Eng, Charlie (id=30) matches Sales. Bob (id=20) and HR (id=40) are dropped.

Auditing a Pipeline: A Worked Example

Suppose you’re analyzing quarterly sales data and you get a result that looks wrong. In a traditional DataFrame library, you’d re-run the pipeline with print statements and hope to catch the problem. In TidyView, you can inspect every intermediate state because nothing was destroyed.

Here’s a concrete scenario. You have eight sales records and you want total revenue by product in the West region:

let sales = Csv.parse("product,region,revenue\nWidget,West,150\nGadget,West,200\nWidget,East,175\nGadget,East,225\nWidget,West,300\nGizmo,East,50\nGadget,West,180\nGizmo,East,75\n");
let view = sales.view();

let west = view.filter(dexpr_binop("==", col("region"), "West"));
let grouped = west.group_by(["product"]);
let summary = grouped.summarise("total_rev", tidy_sum("revenue"));

The summary shows Widget: 450, Gadget: 380. Is that right? Let’s audit backward.

Step 1: Which rows survived the filter?

The bitmask after filter(region == "West") has bits set at positions [0, 1, 4, 6]. That’s four rows out of eight. You can verify: row 0 is Widget/West/150, row 1 is Gadget/West/200, row 4 is Widget/West/300, row 6 is Gadget/West/180. All West. None of the East rows slipped through.

Step 2: How did the groups form?

The group index shows two groups in first-occurrence order:

  • Group “Widget” → base-frame rows [0, 4]
  • Group “Gadget” → base-frame rows [1, 6]

Notice: Gizmo isn’t here. Both Gizmo rows were East, so they were filtered out before grouping. If you expected Gizmo to appear, the bitmask tells you exactly why it didn’t — rows 5 and 7 have their bits cleared.

Step 3: Verify the aggregation

Widget revenue: row 0 (150) + row 4 (300) = 450. Gadget revenue: row 1 (200) + row 6 (180) = 380. Both use Kahan compensated summation, so even with millions of rows the result would be bit-identical across runs.

Step 4: Is the original data intact?

Yes. view.nrows() still returns 8. You can re-filter, re-group, or inspect any column at any time. If the final number is wrong, you trace backward through bitmasks and group indices until you find where your understanding diverged from reality.

Catching Mistakes at Compile Time

TidyView’s type system and error reporting catch common mistakes before they become wrong numbers:

Typo in a column name:

view.filter(dexpr_binop(">", col("revnue"), 100));
// → Error: column `revnue` not found

Self-referencing mutate:

view.mutate("bonus", dexpr_binop("*", col("salary"), 0.1))
    .mutate("double_bonus", dexpr_binop("*", col("bonus"), 2));

The second mutate sees bonus because it’s a separate call on the result of the first. But if you tried to reference a column in the same mutate call that creates it:

view.mutate("a", dexpr_binop("+", col("x"), 1),
            "b", dexpr_binop("*", col("a"), 2));
// → Error: column `a` not found

This is snapshot semantics — all column references are validated against the column list frozen before any assignments. You can’t accidentally create an order-dependent pipeline within a single operation.

Type mismatch:

view.filter(dexpr_binop("+", col("name"), 1));
// → Error: type mismatch — expected Bool predicate, got Int

The filter predicate must evaluate to Bool. Arithmetic on a string column, or a predicate that returns a number instead of true/false, is caught immediately with an error that names the exact types involved.

Benchmark Results

The claims above — bitmask filtering is cheap, chained filters don’t copy data, determinism holds across runs — are testable. I wrote a 38-test benchmark suite that measures TidyView’s performance and verifies its architectural guarantees. Here are the results, run on my machine in release mode.

A note on context. These benchmarks measure TidyView against itself — lazy vs. eager, scaling behavior, determinism proofs. They are not comparative benchmarks against Polars, DuckDB, or Pandas. TidyView is single-threaded and prioritizes determinism over throughput; it will be slower than production-optimized libraries on raw performance. The goal here is to verify that TidyView’s architectural claims hold, not to claim speed parity with tools that make different trade-offs.

A. Filter Microbenchmarks

Test Lazy (µs) Eager (µs) Ratio
Int filter, 100K rows 80 6,459 80×
Float filter, 100K rows 94 6,322 67×
String filter, 100K rows 9,821 13,496 1.4×
Int filter, 1M rows 971 63,605 66×

What this proves: Lazy bitmask filtering is 65–80× faster than eager materialization for numeric columns. The filter itself just scans one column and sets bits — no row copying, no allocation beyond the bitmask. String filters are slower because string comparison is inherently more expensive than integer comparison, but the lazy/eager gap still exists.

The 1M-row int filter takes under 1 ms to produce a bitmask. Materializing — actually copying the surviving rows into a new DataFrame — takes 64 ms. If you’re chaining multiple filters, you save that materialization cost at every step.

B. Chained Filter Composition

Test Time (µs)
Single int filter, 100K 96
Two chained int filters, 100K 175
Int + string chain, 100K 7,210
Two string filters, 100K 12,821

What this proves: Chaining two filters costs roughly the sum of the individual filters, not more. The second filter reads through the first filter’s bitmask — it only examines rows where the bit is set. No intermediate DataFrame is created. The cost of the chain is dominated by the most expensive predicate type (string comparison).

C. Group-By + Summarise

Test Groups Time (µs)
Count, 4 groups, 100K 4 11,473
Mixed agg (5 stats), 4 groups, 100K 4 13,042
Count + Sum, 100 groups, 100K 100 18,073
Count, 10K groups, 100K 10,000 35,930
Count + Sum + Mean, 10K groups, 100K 10,000 39,823

What this proves: Group-by cost scales with the number of distinct groups, not just row count. With 4 groups (the region column), grouping 100K rows takes about 11 ms. With 10K groups, it takes 36 ms. This is the cost of the BTreeMap-based index — O(N log G) where G is the number of groups. Adding more aggregation functions (from Count alone to Count + Sum + Mean + Sd + Median) adds relatively little overhead; the dominant cost is building the group index, not computing the aggregates.

D. Sorting

Test Time (µs)
Float descending, 100K 32,972
Int ascending, 100K 37,072
String ascending, 100K 66,787
Multi-key (string + float), 100K 89,806

What this proves: Sorting is the most expensive single operation because it must materialize the result (you can’t lazily represent a sort). String sorting is roughly 2× more expensive than numeric sorting due to comparison cost. Multi-key sorting adds the cost of a secondary comparator. All sorts use Rust’s stable sort (sort_by), which preserves the relative order of equal elements — important for determinism.

E. Joins

Test Time (µs)
Inner join, 100K × 100 34,063
Inner join, 100K × 1K 38,998
Left join, 100K × 100 30,813

What this proves: Join performance is dominated by the left table size. Increasing the right table from 100 to 1,000 rows adds only ~14% to the join time, because the right table is indexed and lookups are fast. Left joins are slightly faster than inner joins in this case because the left join doesn’t need to check for unmatched right-side rows.

F. Full Pipeline Benchmarks

Pipeline Time (µs)
filter → group → summarise → arrange, 100K 12,739
filter → mutate → group → summarise, 100K 23,819
join → filter → group → summarise, 100K 48,316

What this proves: Realistic multi-step pipelines compose predictably. The filter-group-summarise-arrange pipeline (12.7 ms) is faster than the sum of its parts because the filter reduces the number of rows that downstream operations process. The join-based pipeline is slowest because joins are the most expensive operation and produce a new materialized DataFrame.

Filter time scales linearly with row count — exactly what you’d expect from a single-pass scan. 10× more rows ≈ 10× more time.

Group-by also scales linearly, but with a higher constant factor due to the BTreeMap insertions.

What this proves: Filter time is nearly constant regardless of table width — because filtering only touches the predicate column, not the others. Materialization scales linearly with column count because it has to copy every column. This is the columnar storage payoff: a table with 101 columns filters as fast as a table with 3 columns.

The One-Sentence Summary

TidyView is a database engine wearing a tidyverse costume, designed so that same data + same code = same result on every machine, every run, every time.

It combines familiar dplyr verbs (filter, select, mutate, group_by, summarise) with database-engine internals (columnar storage, bitmask filtering, predicate pushdown, zone maps) to deliver data pipelines that are deterministic by construction, auditable by design, and memory-safe without runtime surprises.

CJC-Lang is open source and available on crates.io. The TidyView implementation lives in crates/cjc-data/ — about 11,000 lines of Rust with zero external dependencies. The full benchmark suite (38 tests, 10 categories) is in tests/tidyview_benchmarks.rs.
