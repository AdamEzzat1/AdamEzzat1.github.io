TL;DR

ABNG is an experimental addition to CJC-Lang shipping in the cjc-abng crate (Phase 0.1 → 0.5). It exposes 75 abng_* builtins for building adaptive belief network graphs — Bayesian-flavored ML primitives that combine codebook-routed BLR posteriors, an SHA-256 audit chain, structural-decision triggers, and per-node provenance stamps. This post walks through eleven distinct capabilities, each proven by a working demo (six of them written in CJC-Lang source and parity-tested across both executors). It also names what’s not proven yet — and why interpreter / compiler work is on the critical path before ABNG becomes production-grade.

ABNG is experimental

ABNG is a research-grade addition to CJC-Lang. The capabilities described here are demonstrated by tests, but not yet productized: there are no benchmarks against industry baselines, no cross-platform CI verification of the determinism canaries, and no real-world case study of someone using ABNG to attest a deployed model. Treat this as a progress report, not a production recommendation. The “what’s still missing” section at the end is honest about the gaps.

What is ABNG?

ABNG — Adaptive Belief Network Graph — is a cjc-abng crate that lives alongside the existing CJC-Lang workspace. The goal: provide ML primitives that go beyond “fit some weights to data” by adding three orthogonal axes:

  1. Bayesian uncertainty at every prediction (mean, epistemic leverage, aleatoric variance) — without MCMC, ensembles, or variational inference.
  2. Cryptographic provenance of training history — every observation, every weight update, every structural decision lands in a SHA-256 audit chain that detects post-hoc tampering.
  3. Adaptive structure — the graph’s topology genuinely changes under the workload. Nodes grow, split, merge, prune, compress, freeze under a deterministic decision-policy engine.

The architecture is a radix-style graph: a root node descends into children indexed by prefix bytes from a quantile codebook, and each leaf carries a Bayesian Linear Regression posterior over a feature basis. Predictions descend the tree, the posterior at the resolved leaf gives (mean, epistemic_leverage, aleatoric_var), and structural decisions fire periodically to adapt the tree.

flowchart LR
    F["State<br/>features"] --> CB["Codebook<br/>encode"]
    CB --> P["Prefix<br/>bytes"]
    P --> D["Descend<br/>tree"]
    D --> L["Leaf<br/>BLR"]
    L --> Pred["(mean,<br/>leverage,<br/>aleatoric)"]
    L -.->|every step| AC["Audit chain<br/>SHA-256"]

ABNG conceptual flow. State features → codebook prefix → descend tree → BLR posterior at leaf → (mean, leverage, aleatoric) — plus an audit chain that hashes every step.

It’s exposed to CJC-Lang as 75 abng_* builtins — abng_new, abng_set_codebook, abng_set_leaf_head, abng_set_blr_prior, abng_observe, abng_blr_predict, abng_descend, abng_decide_step, abng_stamp_provenance, etc. — so you can build and drive an ABNG graph entirely from .cjcl source.

The proof discipline

Every capability in this post has the same proof structure:

  1. A workload — a small but non-trivial problem where the capability should manifest if it’s working.
  2. A specific assertion that would fail if the capability didn’t work.
  3. A locked SHA-256 canary of the resulting graph’s chain_head, gating any future regression.
  4. Cross-pipeline determinism: most demos run through both cjc-eval (AST tree-walk) AND cjc-mir-exec (MIR register-machine) with byte-identical printed output asserted.

That last property is the strongest determinism gate the project supports. AST and MIR are completely different code paths — if any FP arithmetic, hash computation, or dispatch routing differs anywhere in the pipeline, the printed output diverges and the parity test fails.

OK — let’s walk through what’s actually proven.

Capability 1 — Per-state Bayesian uncertainty

Demo: PINN heat-equation solver (tests/test_abng_pinn_uncertainty_cjcl.rs + pinn_source.rs)

What this demo proves

The Bayesian Linear Regression posterior at each codebook-routed leaf returns a 3-tuple (mean, epistemic_leverage, aleatoric_var). The headline assertion is a strict spatial-partition property:

min(epistemic_leverage at edge probes) > max(epistemic_leverage at interior probes)

This is stronger than “average edge uncertainty exceeds average interior uncertainty” — it’s a no-overlap claim. After training, the worst-case interior point has lower uncertainty than the best-case edge point. The signal cleanly partitions evidence-rich vs evidence-poor regions of the input space.

The fit-quality assertion (abs_err_50 < 0.05) proves the mean also converges to the analytical solution. Without this, “leverage is uncertain at edges” would be vacuous — the mean could just be wrong everywhere.

Why the heat equation

The 1-D heat equation ∂u/∂t = α∂²u/∂x² with Dirichlet BCs u(0,t) = u(1,t) = 0 and IC u(x,0) = sin(πx) was chosen because it has a known closed-form analytical solution: u(x,t) = exp(-α π² t) · sin(πx).

That choice is load-bearing for the demo. The BLR feature basis is [1, x, sin(πx), cos(πx)] — the analytical solution exp(-π²·0.1)·sin(πx) is exactly representable as a single non-zero coefficient on the sin(πx) basis function. So with enough samples, the BLR posterior mean must converge to the truth. If it doesn’t, the BLR implementation is broken, not the test.

The training schedule is deliberately asymmetric: 32 samples uniformly across [0.25, 0.75] (interior), 4 samples each in [0, 0.25) and [0.75, 1] (edges). Without this asymmetry, uniform training would give uniform uncertainty and the spatial-partition property would be invisible.

What this enables

  • Active learning in scientific computing — query the BLR for uncertainty; sample where uncertainty is highest.
  • Uncertainty-aware PINNs — refuse to predict in regions where leverage exceeds a threshold; fall back to a higher-fidelity solver.
  • Uncertainty propagation — feed (mean ± sqrt(aleatoric_var + epistemic_leverage)) into Monte Carlo / robust optimization pipelines.
  • Calibrated predictions out of the box — vanilla MLPs don’t say “I don’t know this region”; ABNG does, and you can route around it.

The PINN literature has spent years adding Bayesian uncertainty to neural PDE solvers via MCMC, variational inference, or deep ensembles. ABNG provides this for free via the BLR posterior — no inference machinery beyond a closed-form Bayesian update.

Capability 2 — Per-leaf bounded scaling (GP-like, scalable)

Demo: Tabular regression on the unit square (tests/test_abng_tabular_gp_cjcl.rs + tabular_source.rs)

What this demo proves

Five distinct properties asserted as separate tests:

  1. max(per_leaf_n_seen) < 0.9 × N — no single leaf holds the entire dataset. Evidence is partitioned, not duplicated.
  2. sum(per_leaf_n_seen) == N — every training point routes to exactly one leaf. No double-counting; no points lost.
  3. max_per_leaf_bounded — the partitioning works as designed. (This is the scalability claim’s empirical foundation.)
  4. trained_mse < 0.5 × prior_mse — the BLR posterior actually learns. Without this, all the structure-of-the-graph properties would be vacuous.
  5. lev_shrinks_with_data — leverage at a fixed probe drops from a 16-pt-trained leaf to a 64-pt-trained leaf. GP-like uncertainty contraction reproduced.

Why bilinear regression on the unit square

The target is y = 0.3 + 0.5x₁ + 0.4x₂ + 0.7·x₁·x₂. Three deliberate choices:

  • Exactly fittable in the BLR feature basis [1, x₁, x₂, x₁·x₂]. With enough samples, BLR posterior mean → exact coefficients [0.3, 0.5, 0.4, 0.7]. This makes the MSE-reduction test meaningful: any residual MSE comes from sample efficiency, not from the model class being wrong.
  • 2-D input with a routing dimension. The codebook routes by x₁ (4 bins) and lets x₂ become a within-leaf feature. This mirrors the canonical ABNG pattern: use one dimension to partition the input space, regress on the rest within each leaf.
  • Unit square domain — predictable, no scaling tricks. Codebook boundaries [0.25, 0.5, 0.75] evenly partition [0, 1].

The 200-point training set is small enough to run quickly through the interpreter, large enough that the per-leaf BLR posteriors clearly converge.

What this enables

The classical Gaussian Process scales O(n³) in training data because the kernel matrix is dense. At n ≈ 10⁴ training points a single GP fit takes minutes-to-hours; at n ≈ 10⁵ it’s infeasible without specialized approximations.

ABNG’s per-leaf BLR is O(d²·n_leaf) per update, where d is the BLR feature dim (4 in this demo) and n_leaf ≪ n. Aggregate cost across N points: O(N · d²) — linear in N.

The trade-off: ABNG’s leaves are independent, so you lose the global covariance structure a GP captures. For workloads where local Bayesian behavior is sufficient (most tabular ML), this is a feature, not a bug — and it’s what enables linear scaling.

Warning

The asymptotic claim is correct by construction (the math says so) but not yet benchmarked against a classical GP at realistic sizes. Before treating ABNG as a “GP killer,” wall-clock benchmarks at n ∈ {10³, 10⁴, 10⁵} need to be run.

Capability 3 — Tamper-evident training history (audit chain)

Demo: Universal property — every demo asserts it.

What every demo proves

Every state mutation in ABNG funnels through one chain step:

new_hash = sha256(previous_hash ‖ canonical_payload(event))

Every demo asserts verify_chain == Ok after training. For an n-event audit log, verify_chain walks the chain forward, recomputing new_hash at every step, and compares to the stored value. Any single byte changed in any payload would fail the recomputation at that event AND every subsequent event (cascade property).

Across the 11 demos, this code path runs hundreds of times. The fact that verify_chain == Ok after every demo’s training session is the empirical proof the audit chain is sound under realistic workloads.

What this enables

  • Forensic post-incident analysis — “Was this model trained on the bad data?” Replay the chain, check.
  • Regulatory replay (FDA, SEC, HIPAA) — the model’s full training history is one cryptographically-linked blob.
  • Reproducibility — papers ship (model.snap, dataset) pairs; reviewers reproduce + verify chain.
  • ML supply chain — fine-tuning extends the chain; provenance of every weight is traceable.

This is closer to Git’s content-addressed model than to traditional ML metadata stores. MLflow / Weights & Biases give you external logs you have to trust; ABNG gives you a model artifact whose history is cryptographically self-attesting.

Capability 4 — Cryptographic dataset attestation

Demo: Lineage attestation with synthetic clinical-trial data (tests/test_abng_lineage_attestation_cjcl.rs + lineage_source.rs)

What this demo proves

This is the strongest single capability in ABNG. The headline test (lineage_attestation_three_signals_independent) proves that tampering one row of a 64-row dataset (changing patient #17’s response from 0.525 to 0.999) produces three independent SHA-256 divergences:

  1. chain_head divergence — every audit event’s hash cascades from row 17 onward.
  2. BLR state_hash divergence — the leaf where the tampered row routes has a different posterior.
  3. provenance_stamp divergence — the dataset fingerprint differs by construction.

The “independent” qualifier is doing real work. Even if an attacker forged one signal (e.g., recomputed the chain to hide the change), the other two would still surface the tamper. Three independent failure modes is a stronger security claim than three correlated ones.

Why a synthetic clinical-trial workload

  • Maps directly to a real industry threat model. “Lab A trains a model claiming approved dataset; attacker substitutes contaminated dataset” is the FDA model-attestation problem in one sentence. The two-graph structure (g_a trained on dataset_a, g_b on tampered dataset_b) literally executes this scenario.
  • 64-row size is realistic for clinical-trial cohorts. Phase II trials often have <200 patients per arm. Tampering a single patient’s outcome (1.5% of the dataset) is the minimum-effort attack.
  • Cubic dose-response curve y = 0.2 + 0.6·dose + 0.1·dose² is biologically plausible AND exactly fittable in the BLR feature basis [1, dose, dose², dose³], giving us ground-truth assertions in addition to the security claims.

The “tamper at row 17” choice is intentional: row 17’s dose=0.5 falls right at a codebook bin boundary. This routes the tampered observation to a specific leaf, making the BLR state_hash divergence clearly attributable to one node’s posterior.

What this enables

Use case Concrete flow
FDA model attestation (pharma) Lab runs trial → trains ABNG → stamps with sha256(approved_dataset). Regulator: cjcl abng explain --model X.snap pred.snap checks all three signals
Financial model audit (SEC, Basel III) “Prove this risk model used only data through model-validation cutoff”
HIPAA-compliant healthcare ML Patient-consent boundaries enforced cryptographically: stamp = sha256(consent_set_v2)
GDPR Article 22 right-to-explanation “Show me the audit trail of training data that produced this decision about me”
Academic reproducibility paper.pdf + model.snap + dataset.csv — anyone with all three can verify the paper’s claims byte-for-byte
Insider-threat in ML supply chain Attacker who fine-tunes a base model on contaminated data CAN’T hide the audit-chain extension

Most ML provenance tools today are external: file-system audit logs, MLflow metadata, sigstore-style attestations. ABNG bakes provenance inside the model artifact. The closest analog isn’t an ML tool — it’s Git’s content-addressed history.

Capability 5 — OOD detection composite

Demo: Asymmetric codebook with deliberately-missing children (tests/test_abng_ood_detection_cjcl.rs + ood_source.rs)

What this demo proves

The headline assertion is three-tier separation:

max(ood_score in dense bins)  <  ood_score in sparse bin  <  ood_score in routing-fall-off bin

Three evidence regimes are cleanly partitioned. The composite score is a function of three independent inputs:

  • density_score — Mahalanobis-style distance from training points
  • epistemic_z — BLR’s leverage at the query phi
  • prefix_distance — (prefix_max − matched_prefix) / prefix_max

The demo exercises all three by construction: dense bins have low density_score AND low epistemic_z. The sparse bin has high density_score (few training points). The bin with no child has high prefix_distance (descent fell off the tree).

Why deliberately omit one child

A symmetric setup (children in all 4 bins, varying training density) would only exercise the density and epistemic signals — not prefix_distance. By deliberately not adding a child for bin 3, queries in bin 3 descend with matched_prefix=0 and the prefix-distance contribution to the composite fires.

The demo is asserting the composite works, not just individual signals. If it only varied density, you’d be left wondering whether the routing-distance contribution is real or just a docstring claim. The asymmetric child set proves it’s real.

What this enables

  • Self-driving / robotics — refuse to act on inputs unfamiliar to the training distribution; hand off to safe-mode.
  • Medical diagnosis — don’t predict on patient profiles outside the training population.
  • Fraud detection — novel attack patterns trigger high OOD scores.
  • Active learning — high-OOD inputs are highest-value labeling candidates.

The ABNG composite is more nuanced than typical “OOD detection” approaches because it composes three orthogonal signals. A query can be OOD because (a) it’s far from training points (density), (b) the model is uncertain there (epistemic), or (c) it doesn’t fit the codebook structure at all (prefix). Each tells a different operational story.

Capability 6 — Adaptive structural triggers

Demo: Two-sibling shared-signature topology (tests/test_abng_adaptive_triggers_cjcl.rs + adaptive_source.rs)

What this demo proves

After running abng_decide_step three times on a graph with two sibling children:

  1. merge_count == 1 — exactly one Merge action fires (deterministic given the canary thresholds).
  2. audit_log_grew — the structural mutation is recorded as an event in the chain.
  3. verify_chain == Ok — the chain integrity holds after structural mutation.
  4. chain_head matches the Rust-side decide_step_canary BYTE-FOR-BYTE — this is the most striking property in the entire demo set.

That last point deserves emphasis. The Rust-side decide_step_canary_tests.rs locks chain_head = d064fb08…7807. The CJC-Lang demo, running through both AST eval AND MIR exec, produces the same hex. The same workload, expressed three different ways (direct Rust API, CJC-Lang AST, CJC-Lang MIR), produces bit-identical SHA-256 output through 23+ events.

Why the two-sibling shared-signature topology

  • It’s the canonical canary setup. Mirrors the Rust-side decide_step_canary_tests exactly — root + 2 children at keys 1 and 2, three observations at root, three decide_step calls. This lets the CJC-Lang demo serve as a direct cross-pipeline determinism gate.
  • Merge is the most reliably-triggered action. With identical training distributions on the two siblings, their signatures align. Decision-policy thresholds tau_merge=0 (Hamming distance) and small kl_merge are satisfied immediately. Merge fires deterministically.
  • Small graph keeps the test fast — 4 nodes, 3 observations, 3 decide_step calls. Runs in milliseconds even through the AST interpreter.

What this enables

  • Self-organizing ML models — graph capacity scales with workload complexity.
  • Streaming / online learning — non-stationary distributions cause Grow on new modes, Prune on dead branches.
  • Memory-efficient ML — Compress unused subtrees; Merge redundant leaves; Prune inactive nodes.
  • Continual learning — Freeze stable-signature subtrees while training others.
Warning

Honest caveat: the demo proves Merge fires deterministically. The other five trigger types (Grow, Split, Prune, Compress, Freeze) are tested at the primitive level but not demonstrated firing in a realistic workload. The “graph adapts to workload complexity” claim is half-substantiated by current demos. Closing this gap is straightforward (more demos with engineered workloads that fire each trigger type) but not yet shipped.

Capability 7 — Calibration / ECE

Demo: Hand-constructed well-calibrated vs miscalibrated outcomes (tests/test_abng_calibration_cjcl.rs + calibration_source.rs)

What this demo proves

Two graphs receive 30 calibration observations each. Same predicted probabilities (p ∈ {0.1, 0.5, 0.9}, 10 each), different outcome distributions:

Graph Outcome convention Expected ECE
A: well-calibrated predicted=0.1 → 1 of 10 positive; predicted=0.9 → 9 of 10 positive < 0.05
B: miscalibrated predicted=0.1 → 9 of 10 positive (FLIPPED); predicted=0.9 → 1 of 10 positive > 0.5

The headline assertion ece_a < ece_b separates the two cleanly. Plus the specific bounds: ece_a < 0.05 (well-calibrated achieves near-zero ECE) and ece_b > 0.5 (flipped outcomes produce huge ECE).

Why “flip outcomes” as the miscalibration model

  • Flipping is the maximally-miscalibrated case. Predicting 90% confidence and being right 10% of the time is the worst possible outcome — confidence is anti-correlated with truth.
  • Same prediction distribution → isolates the calibration signal. Both graphs see the same (p, y) pairs modulo y. Any difference in ECE is attributable to outcomes, not predictions.
  • Three reliability bins (0.1, 0.5, 0.9) span low/mid/high probability range without requiring 100+ observations to populate every bin.

What this enables

  • Risk-sensitive deployment — a poorly-calibrated 95%-confident prediction is operationally indistinguishable from an honest 60%-confident one — but you’ll act on it differently.
  • Threshold tuning — “should I act on predictions above 0.7?” depends on whether 0.7-bin predictions are actually correct 70% of the time.
  • Model selection in production — beyond accuracy/F1: ECE is the second-order quality metric.
  • Active learning targeting — high-ECE bins are evidence the model needs more training data in that probability range.

Capability 8 — Distribution-drift detection

Demo: Mean shift from 0 to 0.8 on a 4-D density tracker (tests/test_abng_drift_detection_cjcl.rs + drift_source.rs)

What this demo proves

A three-phase workload: train baseline, freeze, observe under same distribution, observe under shifted distribution. Three assertions:

  1. drift_grew — drift_score(post-shift) > drift_score(same-dist).
  2. meaningful_shift — drift_score(post-shift) > 2 × drift_score(same-dist). Catches a regression where drift detection becomes noisy.
  3. drift_nonneg — both scores are non-negative (no NaN, no negative noise).

Why a pure mean shift

  • Translation, not rescaling. The Mahalanobis-style distance the density tracker uses decomposes into mean-shift + covariance-shift contributions. By shifting only the mean, the test isolates that contribution cleanly.
  • 0 to 0.8 is large enough to be unambiguous. Both distributions have ~0.01 jitter (effectively delta functions for density purposes); shifting the center by 0.8 is 80x the jitter — the drift signal saturates well above the floor.
  • Same density-tracker geometry. Both phases use the same dimension count (4-D features), same covariance scale, same batch sizes (16 each post-baseline). Only the location changes — isolating “distribution shifted” from confounders.

What this enables

  • Production ML monitoring — alert ops when input distribution drifts from training; trigger retraining.
  • Continual learning auto-trigger — drift signal is the cue to incorporate new data into the model.
  • A/B test data validation — confirm the new dataset’s distribution matches the original before deploying a new model.
  • Sensor / pipeline integrity — detect upstream data-source switches, sensor drift, schema changes.

Capability 9 — Log compaction

Demo: Multi-node observation stream + compact_log (tests/test_abng_compact_log_cjcl.rs + compact_source.rs)

What this demo proves

Five training observations across three nodes (root + 2 children, each receives 1-2 observations), then compact_log fires:

  1. emitted == 3 — exactly one StatsSnapshot per touched node.
  2. audit_grew_by_emitted — audit_post == audit_pre + emitted.
  3. chain_advanced — chain_head changed (events were properly chain-linked).
  4. verify_chain == Ok post-compact — the new events don’t break integrity.

What this enables

  • Long-running training without log explosion — periodic compaction preserves checkpoints while bounding audit-log growth.
  • Snapshot-based debugging — each StatsSnapshot is a “save point” for forensic replay.
  • Foundation for the Phase 0.6 fast-forward optimization — smart-replay can fast-forward past observation runs by trusting the StatsSnapshot.
Warning

The full read-side optimization is deferred to a future phase. Today the marker layer ships with chain-integrity preservation; the cycle-saving “skip the observe” optimization that would make this user-visibly faster is Phase 0.6 work.

Capability 10 — Maturity observability

Demo: Stable observation stream + decide_step (tests/test_abng_maturity_inspection_cjcl.rs + maturity_source.rs)

What this demo proves

Read abng_node_maturity at two timepoints: fresh graph (T=0) and after 10 stable observations + 5 decide_step calls (T=1). Five assertions:

  1. sum_at_t0 == 0.0 — fresh graph, all maturity flags are zero.
  2. maturity_evolved — sum at T=1 strictly exceeds sum at T=0.
  3. calib_binary AND unc_binary — positions [1] (calibration_stable) and [2] (uncertainty_stable) are strict 0/1 flags as documented.
  4. samples_seen_nonneg — position [0] (samples_seen) is non-negative.
  5. trust_in_range — position [3] (trust_level) is in [0, 1].

The maturity tensor turns out to be heterogeneous — a count, two binary flags, and a continuous trust level. Building this demo is what flushed out the API quirk; the docstring on abng_node_maturity describes it correctly but the behavior is easy to mis-assume.

What this enables

  • Production ML monitoring dashboard — “has my model converged?” → check maturity per-node.
  • Auto-stop training — stop when maturity flags have all stabilized.
  • Convergence-failure debugging — which flag never reaches 1? — points at the failing component.
  • Production rollout gating — deploy-time check: only ship models where calibration_stable == 1.

Capability 11 — End-to-end determinism

Demo: Universal property — every CJC-Lang demo asserts it.

What every demo proves

Every CJC-Lang demo runs through both cjc-eval (AST tree-walk) AND cjc-mir-exec (MIR register-machine), then asserts byte-equal printed output. Across the 13 ABNG demo files, this constitutes ~95 byte-equal parity assertions over a vocabulary of ~30 abng_* builtins.

The strongest single instance: the adaptive triggers demo’s chain_head matches the Rust-side decide_step_canary byte-for-byte (d064fb08…7807). Same workload, three pipelines (direct Rust API + AST eval + MIR exec), bit-identical SHA-256 over 23+ audit events.

Why this is hard to deliver in practice

Three architectural choices that make this possible:

  • All FP arithmetic uses Kahan compensation — no FMA, no SIMD-introduced reordering, deterministic add-order semantics.
  • All maps are BTreeMap — iteration order is sorted, never random. HashMap with random hashing would silently break determinism.
  • All RNG is SplitMix64 with explicit seed threading — no thread-local randomness, no time-based seeds, no platform-specific entropy.

Most ML stacks claim “deterministic with same seed” but produce subtly different bytes across runs. ABNG’s parity gates prove determinism every test run.

Warning

Cross-platform determinism is not yet verified. All canary hashes were computed on Windows. They should hold on Linux/macOS (Kahan, no FMA, BTreeMap), but this isn’t yet tested in CI.

What’s next — and why interpreter / compiler work matters

ABNG is genuinely useful for a specific niche: regulated ML training where you need Bayesian uncertainty + cryptographically-verifiable lineage on tabular data. The demos prove the niche capabilities; they don’t prove ABNG should replace PyTorch.

Three things stand between current ABNG and production readiness:

Cross-platform determinism CI

The locked canary hashes are Windows-only today. They should hold across platforms (the architecture is designed for it), but this is unverified. Adding GitHub Actions CI that runs the full test suite on Linux + macOS + Windows and compares the canaries is a one-day project that pays off forever — every future commit either preserves cross-platform determinism or fails CI loudly.

Performance benchmarks

The “linear scaling” claim for ABNG vs classical GP is correct asymptotically but not yet benchmarked at realistic dataset sizes. Without n ∈ {10³, 10⁴, 10⁵} wall-clock numbers, this is a “promising primitive” not a “GP killer.”

Compiler / interpreter performance

This is the bigger long-term constraint. The CJC-Lang interpreter is currently 5–50× slower than direct Rust API access for ABNG demos. For research / experimentation that’s fine. For production ML training at scale, it’s a hard ceiling.

The chess RL benchmark hit this wall directly — project memory: “bottleneck is interpreter throughput, not ML algorithm design.” Specific levers, in increasing order of effort:

  • MIR optimizer passes — current optimizer has CF + DCE; further passes (loop-invariant code motion, common subexpression elimination, function inlining) are weeks of work each.
  • Hot-path specialization in cjc-runtime — ABNG already shows the pattern (75 native builtins for what users actually do). Has architectural cost: every native kernel is a feature you can’t compose out of simpler primitives.
  • AOT compilation — cjcl compile foo.cjcl → foo.exe. LLVM or Cranelift backend. Months of work but the only path that closes the perf gap meaningfully.

Most likely sequencing: continue ABNG (Phase 0.6 work — smart-replay fast-forward, cross-platform CI, lineage case study) while landing incremental MIR optimizer + native-kernel work. AOT is a multi-month commitment that depends on whether CJC-Lang’s long-term identity is research vehicle or production language.

Honest aggregate

ABNG is a 7/10 system if you have the niche problems it solves, and a 2/10 system if you don’t. The demos honestly bound where it’s useful:

Strong fit (lead with ABNG): - Pharma / financial / healthcare ML with regulatory attestation needs - Academic groups publishing reproducibility-critical results - Tabular Bayesian regression at sizes prohibitive for classical GP

Moderate fit (consider ABNG alongside existing tools): - Production ML monitoring (drift detection + existing observability) - Active learning research where BLR uncertainty is sufficient - Scientific computing groups doing 1-D / 2-D PINNs

Poor fit (don’t reach for ABNG): - Deep learning, vision, NLP, audio — codebook routing doesn’t fit - Real-time inference / latency-sensitive serving — audit-chain overhead - Quick experimentation / prototyping — overkill

The honest one-sentence summary: “ABNG is what you reach for when you need to prove what your model was trained on, with calibrated Bayesian uncertainty, and you can tolerate the audit-chain overhead.”

That’s a real market — but it’s a niche, not the mainstream. The work ahead — interpreter perf, cross-platform CI, real case studies — is what would graduate ABNG from “demonstrably works in tests” to “deployed in production somewhere that matters.”

For now: experimental, but with clear value for the right user. If that’s you — you know who you are — get in touch.
