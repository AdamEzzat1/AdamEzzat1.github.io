TL;DR

ABNG is an experimental addition to CJC-Lang shipping in the cjc-abng crate (Phase 0.1 → 0.5). It exposes 75 abng_* builtins for building adaptive belief network graphs — Bayesian-flavored ML primitives that combine codebook-routed BLR posteriors, an SHA-256 audit chain, structural-decision triggers, and per-node provenance stamps. This post walks through eleven distinct capabilities, each proven by a working demo (six of them written in CJC-Lang source and parity-tested across both executors). It also names what’s not proven yet — and why interpreter / compiler work is on the critical path before ABNG becomes production-grade.

ABNG is experimental

ABNG is a research-grade addition to CJC-Lang. The capabilities described here are demonstrated by tests, but not yet productized: there are no benchmarks against industry baselines, no cross-platform CI verification of the determinism canaries, and no real-world case study of someone using ABNG to attest a deployed model. Treat this as a progress report, not a production recommendation. The “what’s still missing” section at the end is honest about the gaps.

What is ABNG?

ABNG — Adaptive Belief Network Graph — is a cjc-abng crate that lives alongside the existing CJC-Lang workspace. The goal: provide ML primitives that go beyond “fit some weights to data” by adding three orthogonal axes:

  1. Bayesian uncertainty at every prediction (mean, epistemic leverage, aleatoric variance) — without MCMC, ensembles, or variational inference.
  2. Cryptographic provenance of training history — every observation, every weight update, every structural decision lands in a SHA-256 audit chain that detects post-hoc tampering.
  3. Adaptive structure — the graph’s topology genuinely changes under the workload. Nodes grow, split, merge, prune, compress, freeze under a deterministic decision-policy engine.

The architecture is a radix-style graph: a root node descends into children indexed by prefix bytes from a quantile codebook, and each leaf carries a Bayesian Linear Regression posterior over a feature basis. Predictions descend the tree, the posterior at the resolved leaf gives (mean, epistemic_leverage, aleatoric_var), and structural decisions fire periodically to adapt the tree.

flowchart LR
    F["State<br/>features"] --> CB["Codebook<br/>encode"]
    CB --> P["Prefix<br/>bytes"]
    P --> D["Descend<br/>tree"]
    D --> L["Leaf<br/>BLR"]
    L --> Pred["(mean,<br/>leverage,<br/>aleatoric)"]
    L -.->|every step| AC["Audit chain<br/>SHA-256"]

ABNG conceptual flow. State features → codebook prefix → descend tree → BLR posterior at leaf → (mean, leverage, aleatoric) — plus an audit chain that hashes every step.

It’s exposed to CJC-Lang as 75 abng_* builtins — abng_new, abng_set_codebook, abng_set_leaf_head, abng_set_blr_prior, abng_observe, abng_blr_predict, abng_descend, abng_decide_step, abng_stamp_provenance, etc. — so you can build and drive an ABNG graph entirely from .cjcl source.

The proof discipline

Every capability in this post has the same proof structure:

  1. A workload — a small but non-trivial problem where the capability should manifest if it’s working.
  2. A specific assertion that would fail if the capability didn’t work.
  3. A locked SHA-256 canary of the resulting graph’s chain_head, gating any future regression.
  4. Cross-pipeline determinism: most demos run through both cjc-eval (AST tree-walk) AND cjc-mir-exec (MIR register-machine) with byte-identical printed output asserted.

That last property is the strongest determinism gate the project supports. AST and MIR are completely different code paths — if any FP arithmetic, hash computation, or dispatch routing differs anywhere in the pipeline, the printed output diverges and the parity test fails.

OK — let’s walk through what’s actually proven.

Capability 1 — Per-state Bayesian uncertainty

Demo: PINN heat-equation solver (tests/test_abng_pinn_uncertainty_cjcl.rs + pinn_source.rs)

What this demo proves

The Bayesian Linear Regression posterior at each codebook-routed leaf returns a 3-tuple (mean, epistemic_leverage, aleatoric_var). The headline assertion is a strict spatial-partition property:

min(epistemic_leverage at edge probes) > max(epistemic_leverage at interior probes)

This is stronger than “average edge uncertainty exceeds average interior uncertainty” — it’s a no-overlap claim. After training, the worst-case interior point has lower uncertainty than the best-case edge point. The signal cleanly partitions evidence-rich vs evidence-poor regions of the input space.

The fit-quality assertion (abs_err_50 < 0.05) proves the mean also converges to the analytical solution. Without this, “leverage is uncertain at edges” would be vacuous — the mean could just be wrong everywhere.

The math

ABNG’s BLR uses a Normal-Inverse-Gamma (NIG) conjugate prior on (w, σ²). Given a prior (μ₀, Λ₀, a₀, b₀) and a stream of observations D = {(φᵢ, yᵢ)}, the conjugate update yields a closed-form posterior:

\[ \begin{aligned} \Lambda_n &= \Lambda_0 + \Phi^\top \Phi \\ \mu_n &= \Lambda_n^{-1}\,(\Lambda_0 \mu_0 + \Phi^\top y) \\ a_n &= a_0 + n/2 \\ b_n &= b_0 + \tfrac{1}{2}(y - \Phi\mu_n)^\top(y - \Phi\mu_n) + \tfrac{1}{2}(\mu_0 - \mu_n)^\top \Lambda_0\,(\mu_0 - \mu_n) \end{aligned} \]

For a query point x*, the predictive distribution p(y* | x*, D) is a Student-t. ABNG returns its three sufficient ingredients as:

Output Formula Interpretation
mean \(\mu_n^\top \varphi(x^*)\) Posterior mean prediction (point estimate)
epistemic_leverage \(\varphi(x^*)^\top \Lambda_n^{-1}\,\varphi(x^*)\) Variance contribution from posterior over w (model uncertainty)
aleatoric_var \(b_n / (a_n - 1)\) Observation-noise variance (data uncertainty)

Two cleanly separated uncertainty channels. epistemic_leverage shrinks as ΦᵀΦ accumulates — more data → tighter posterior over w → less model uncertainty. aleatoric_var converges to the irreducible noise floor. A vanilla MLP collapses both into a single point estimate; ABNG keeps them factored.

Why the heat equation

The 1-D heat equation ∂u/∂t = α∂²u/∂x² with Dirichlet BCs u(0,t) = u(1,t) = 0 and IC u(x,0) = sin(πx) was chosen because it has a known closed-form analytical solution:

\[u(x, t) = e^{-\alpha \pi^2 t} \sin(\pi x)\]

That choice is load-bearing. The BLR feature basis is [1, x, sin(πx), cos(πx)] — the analytical solution exp(-π²·0.1)·sin(πx) is exactly representable as a single non-zero coefficient on the sin(πx) basis function. So with enough samples, the BLR posterior mean must converge to the truth. If it doesn’t, the BLR implementation is broken — not the test. This is the same trick statisticians use to validate Bayesian implementations: pick a problem where the truth is in the basis, and the posterior is a closed-form limit.

The training schedule is deliberately asymmetric: 32 samples uniformly across [0.25, 0.75] (interior), 4 samples each in [0, 0.25) and [0.75, 1] (edges). Without this asymmetry, uniform training would give uniform leverage and the spatial-partition property would be invisible.

Code sketch

The PINN demo’s core loop, in CJC-Lang:

fn pinn_features_2d(x: f64) -> Tensor {
    Tensor.from_vec([1.0, x, sin(pi() * x), cos(pi() * x)], [1, 4])
}

fn analytical_u(x: f64) -> f64 {
    let decay = exp(-0.9869604401089358);  // exp(-π² · 0.1)
    decay * sin(pi() * x)
}

fn train_one(g: i64, x: f64) {
    let leaf = route_leaf(g, x);             // codebook descent
    let phi = pinn_features_2d(x);
    let y_val = analytical_u(x);             // ground truth
    let y = Tensor.from_vec([y_val], [1]);
    abng_blr_update(g, leaf, phi, y);        // NIG conjugate update
    abng_observe(g, leaf, y_val);
}

After training, a probe at x = 0.5 returns (mean ≈ 0.372, lev_low, ale_finite) — mean matches exp(-π²·0.1) · sin(π·0.5) = 0.372 to within 0.05. A probe at x = 0.05 returns (mean ≈ small, lev_high, ale_finite) — different leaf, less data, higher leverage.

Connection to existing techniques

Technique Posterior representation Cost Decomposed epistemic/aleatoric?
Gaussian Processes Full posterior over functions O(n³) Cholesky Yes (kernel + likelihood noise)
Deep ensembles Variance across M trained nets M× training cost Approximately
MC dropout Variational approx via dropout Cheap, biased No (entangled)
Conformal prediction Distribution-free intervals Cheap No
ABNG per-leaf BLR Closed-form NIG posterior O(d²) per update Yes (clean factor)

ABNG slots between GPs (full posterior, expensive) and conformal prediction (distribution-free, no decomposition). It’s most similar in spirit to Bayesian Additive Regression Trees (BART) — leaf-localized closed-form Bayesian fits — but with a quantile-codebook routing instead of stochastic tree growth, and a stronger determinism contract.

What this enables

  • Active learning in scientific computing — query the BLR for epistemic_leverage; sample where it’s highest.
  • Uncertainty-aware PINNs — refuse to predict in regions where leverage exceeds a threshold; fall back to a higher-fidelity solver. This is what the PINN community has been chasing via deep ensembles or HMC for years.
  • Heteroskedastic regression done right — the aleatoric/epistemic split lets you tell “this region is intrinsically noisy” from “I haven’t seen enough of this region yet.”
  • Uncertainty propagation downstream — feed the Student-t predictive into Monte Carlo / robust optimization pipelines.

The PINN literature has long treated Bayesian uncertainty as expensive infrastructure (HMC, variational PINNs). ABNG provides this as a closed-form per-leaf posterior — no inference machinery beyond a Cholesky update.

Capability 2 — Per-leaf bounded scaling (GP-like, scalable)

Demo: Tabular regression on the unit square (tests/test_abng_tabular_gp_cjcl.rs + tabular_source.rs)

What this demo proves

Five distinct properties asserted as separate tests:

  1. max(per_leaf_n_seen) < 0.9 × N — no single leaf holds the entire dataset. Evidence is partitioned, not duplicated.
  2. sum(per_leaf_n_seen) == N — every training point routes to exactly one leaf. No double-counting; no points lost.
  3. max_per_leaf_bounded — the partitioning works as designed. (This is the scalability claim’s empirical foundation.)
  4. trained_mse < 0.5 × prior_mse — the BLR posterior actually learns. Without this, the structure-of-the-graph properties would be vacuous.
  5. lev_shrinks_with_data — leverage at a fixed probe drops from a 16-pt-trained leaf to a 64-pt-trained leaf. GP-like uncertainty contraction reproduced.

The math (cost analysis)

ABNG’s per-leaf cost decomposes cleanly:

Operation Cost
Codebook descent O(n_dims · log(n_bins))
BLR conjugate update at one leaf O(d² · n_leaf) (Cholesky-factor maintenance)
Predictive at query point O(d²) (one Cholesky solve)
Aggregate cost across N points, L leaves O(N · d²) — linear in N

Compare to a classical GP on the same N points:

Operation Cost
Kernel matrix K_NN O(N²) memory
Cholesky decomposition O(N³) compute
Predictive variance per query O(N²)

ABNG is asymptotically O(N · d²) vs GP’s O(N³). For d = 4 (the demo’s basis dim) and N = 10⁴: ABNG ≈ 1.6×10⁵ ops; GP ≈ 10¹² ops — seven orders of magnitude.

The trade-off: ABNG’s leaves are independent — no global covariance structure. For workloads where local Bayesian behavior is sufficient (most tabular ML), this is a feature, not a bug.

Why bilinear regression on the unit square

The target is y = 0.3 + 0.5x₁ + 0.4x₂ + 0.7·x₁·x₂. Three deliberate choices:

  • Exactly fittable in the BLR feature basis [1, x₁, x₂, x₁·x₂]. With enough samples, BLR posterior mean → exact coefficients [0.3, 0.5, 0.4, 0.7]. This makes the MSE-reduction test meaningful: any residual MSE comes from sample efficiency, not from the model class being wrong.
  • 2-D input with a routing dimension. The codebook routes by x₁ (4 bins) and lets x₂ become a within-leaf feature. This mirrors the canonical ABNG pattern: use one dimension to partition the input space, regress on the rest within each leaf. Common in production: route by (time-of-day, geo-region) and regress on the rest within each segment.
  • Unit square domain — predictable, no scaling tricks. Codebook boundaries [0.25, 0.5, 0.75] evenly partition [0, 1].

Code sketch

fn build_tabular_graph(seed: i64) -> i64 {
    let g = abng_new(seed);
    let codebook = Tensor.from_vec([0.25, 0.5, 0.75], [1, 3]);
    abng_set_codebook(g, codebook);                          // 4 bins on x₁
    abng_set_leaf_head(g, 2, Tensor.from_vec([4.0], [1]), 1, "tanh");
    abng_set_blr_prior(g, 2.0, 1.0, 0.5);                    // NIG (a, b, σ_init)
    // ... add 4 children, one per codebook bin ...
    g
}

fn train_one(g: i64, x1: f64, x2: f64) {
    let leaf = route_for(g, x1);                             // x₁ → prefix → leaf
    let phi = Tensor.from_vec([1.0, x1, x2, x1 * x2], [1, 4]);
    let y_val = target(x1, x2);
    abng_blr_update(g, leaf, phi, Tensor.from_vec([y_val], [1]));
}

After 200 training points, every leaf has its own BLR posterior with mean → [0.3, 0.5, 0.4, 0.7] and increasingly sharp leverage. Sum of n_seen across all four leaves equals exactly 200; the largest leaf holds well under 90% of points.

Connection to existing techniques

Technique Routing Per-leaf model Scaling
Classical GP None (global) Kernel O(n³)
Sparse GP / SVGP Inducing points Kernel O(n·m²) for m inducing pts
Mondrian Forest Stochastic tree splits Point estimate O(n·log n)
BART Stochastic tree splits Bayesian posterior O(n·log n) per tree
Locally-weighted regression Distance-weighted Linear fit O(n) per query
ABNG Quantile codebook (deterministic) Closed-form NIG posterior O(N · d²)

The closest analog is Mondrian Forests (Lakshminarayanan, Roy & Teh 2014) — both partition the input space and fit per-leaf models. The differences: Mondrian uses stochastic splits and point estimates; ABNG uses deterministic quantile boundaries and full closed-form Bayesian posteriors. ABNG also adds the audit chain that Mondrian doesn’t have.

What this enables

  • Drop-in tabular Bayesian regression at GP-prohibitive sizes (n > 10⁴ where dense Cholesky becomes painful)
  • Online / streaming learning — incremental BLR update is O(d²) per point, vs O(n²) incremental for classical GP rank-one updates
  • Snapshot-checkpointable Bayesian state — classical GPs don’t naturally serialize (you’d persist the full Cholesky factor + raw data); ABNG snapshots the entire posterior in one blob via the audit chain
  • Feature-engineered uncertainty quantification — the practitioner picks routing dimensions (time-of-day, region, customer segment) and the per-segment posteriors are independently calibrated
Warning

The asymptotic claim is correct by construction (the math says so) but not yet benchmarked against a classical GP at realistic sizes. Wall-clock benchmarks at n ∈ {10³, 10⁴, 10⁵} need to be run before treating ABNG as a “GP killer.”

Capability 3 — Tamper-evident training history (audit chain)

Demo: Universal property — every demo asserts it.

What every demo proves

Every state mutation in ABNG funnels through one chain step:

\[ \text{new\_hash} = \text{SHA-256}\bigl(\text{previous\_hash} \,\Vert\, \text{canonical\_payload}(\text{event})\bigr) \]

Every demo asserts verify_chain == Ok after training. For an n-event audit log, verify_chain walks the chain forward, recomputing new_hash at every step from the stored previous hash and the stored canonical payload, and compares to the stored value. Any single byte changed in any payload would fail the recomputation at that event AND every subsequent event — a Merkle-style cascade.

Across the 11 demos, this code path runs hundreds of times. The fact that verify_chain == Ok after every demo’s training session is the empirical proof the audit chain is sound under realistic workloads.

The threat model

The audit chain is designed against three concrete attacks:

  1. Post-hoc rewrite. Attacker modifies a historical training observation after the fact. Detection: the modified event’s new_hash no longer matches sha256(previous_hash ‖ payload), AND every subsequent event’s stored previous_hash no longer matches the recomputed-from-modified value. Cascading detection.
  2. Event reordering. Attacker shuffles the order of training events. Detection: each event’s payload includes its seq number; reordering puts events at the wrong seq, breaking the per-event monotonic-seq invariant.
  3. Event substitution. Attacker replaces one event with another that produces the same new_hash. Detection: SHA-256 collision resistance — finding two payloads with the same hash is computationally infeasible.

Code sketch — what an event payload looks like

[seq u64 BE | epoch u64 BE | node_id u32 BE | kind tag u8 |
 kind body (variable) |
 stats_version u64 BE | stats_hash [u8; 32]]

For a BeliefUpdate event (the most common kind, fired on every abng_observe), the kind body is a single value f64-bits BE — 8 bytes. Total payload for a BeliefUpdate is 21 + 8 + 8 + 32 = 69 bytes, hashed into the chain.

The canonical encoding is byte-deterministic: big-endian on every numeric field, fixed sizes, no padding ambiguity. Two ABNG instances run with the same seed and observation order produce identical event payload bytes — and therefore identical chain hashes.

Connection to existing techniques

Technique Hash-chain ML-aware Per-prediction lineage
Git Yes (DAG of commits) No No
MLflow No (timestamped log) Yes Manual
sigstore / in-toto Yes (artifact attestation) No Per-artifact
Blockchain ML provenance Yes (full chain on-chain) Yes Yes, but expensive
ABNG audit chain Yes (linear) Yes (every state mutation) Yes, baked in

The closest analog is in-toto attestations in software supply-chain land — except in-toto is per-artifact (each release gets one attestation), while ABNG is per-event (every observation, every weight update).

What this enables

  • Forensic post-incident analysis — “Was this model trained on the bad data?” Replay the chain, check.
  • Regulatory replay — the model’s full training history is one cryptographically-linked blob.
  • Reproducibility certification — papers ship (model.snap, dataset) pairs; reviewers reproduce + verify chain in one byte comparison.
  • ML supply chain attestation — fine-tuning extends the chain; provenance of every weight is traceable.
  • Insider-threat detection — anyone modifying historical events breaks every subsequent hash, visible to anyone running verify_chain.

This is closer to Git’s content-addressed model than to traditional ML metadata stores. MLflow / Weights & Biases give you external logs you have to trust; ABNG gives you a model artifact whose history is cryptographically self-attesting.

Capability 4 — Cryptographic dataset attestation

Demo: Lineage attestation with synthetic clinical-trial data (tests/test_abng_lineage_attestation_cjcl.rs + lineage_source.rs)

What this demo proves

The headline test (lineage_attestation_three_signals_independent) proves that tampering one row of a 64-row dataset (changing patient #17’s response from 0.525 to 0.999) produces three independent SHA-256 divergences:

  1. chain_head divergence — every audit event’s hash cascades from row 17 onward.
  2. BLR state_hash divergence — the leaf where the tampered row routes has a different posterior.
  3. provenance_stamp divergence — the dataset fingerprint differs by construction.

The “independent” qualifier is doing real work. Even if an attacker forged one signal (e.g., recomputed the chain to hide the change), the other two would still surface the tamper.

The threat model and why three signals matter

The classic attack against ML provenance is collusion: the attacker controls the training process, so they can produce a consistent-but-fake audit trail. Three independent signals raise the cost:

Attack Defeats which signal? Defeats all three?
Modify training data, recompute chain chain_head ✓ No — BLR state diverges
Modify training data, retrain to match BLR state chain_head ✓, BLR state ✓ No — provenance stamp diverges
Modify training data, retrain, restamp with new fingerprint All three forgeable internally But: fingerprint is publicly verifiable against the dataset bytes

The provenance stamp is sha256(canonical_dataset_bytes). An auditor with the original dataset can recompute this themselves — the attacker can’t forge it without also forging the dataset (which is what we’re trying to detect in the first place). The chain and BLR state are internal consistency checks; the provenance stamp is the external binding to the dataset.

Why a synthetic clinical-trial workload

  • Maps directly to a real industry threat model. “Lab A trains a model claiming approved dataset; attacker substitutes contaminated dataset” is the FDA model-attestation problem in one sentence.
  • 64-row size is realistic for Phase II clinical-trial cohorts (often <200 patients per arm).
  • Cubic dose-response curve y = 0.2 + 0.6·dose + 0.1·dose² is biologically plausible AND exactly fittable in the BLR basis [1, dose, dose², dose³], giving us ground-truth assertions in addition to the security claims.
  • Tamper at row 17 specifically because dose=0.5 falls right at a codebook bin boundary — routing the tampered observation to a specific leaf, making the BLR state_hash divergence clearly attributable.

Code sketch

fn train_and_stamp(seed: i64, rows: Any, fingerprint: String) -> i64 {
    let g = build_lineage_graph(seed);
    abng_stamp_provenance(g, 0, fingerprint);                // sha256(dataset_bytes)
    let n = len(rows);
    let i = 0;
    while i < n {
        let row = rows[i];
        let dose = row[0];
        let resp = row[1];
        let leaf = route_leaf(g, dose);
        let phi = Tensor.from_vec(
            [1.0, dose, dose * dose, dose * dose * dose], [1, 4]);
        abng_blr_update(g, leaf, phi, Tensor.from_vec([resp], [1]));
        abng_observe(g, leaf, resp);
        i = i + 1;
    }
    g
}

The two graphs g_a and g_b are built identically — only the row data and fingerprint differ. Comparing (chain_head, blr_state_hash[node 3], provenance_stamp) between them yields three different hashes.

Capability 5 — OOD detection composite

Demo: Asymmetric codebook with deliberately-missing children (tests/test_abng_ood_detection_cjcl.rs + ood_source.rs)

What this demo proves

The headline assertion is three-tier separation:

max(ood_score in dense bins)  <  ood_score in sparse bin  <  ood_score in routing-fall-off bin

Three evidence regimes are cleanly partitioned. The composite is a function of three independent inputs.

The math

ABNG’s ood_score composites three signals:

\[\text{ood}(x) = f\bigl(\text{density}(x),\ \text{epistemic\_z}(x),\ \text{prefix\_distance}(x)\bigr)\]

where:

Signal Formula What it detects
density(x) \(1 - \exp(-d^2)\) where \(d^2 = (x - \bar{\mu})^\top \text{diag}(\Sigma)^{-1}(x - \bar{\mu})\) Distance from training cluster mean (per-leaf)
epistemic_z(x) \(\min(\varphi^\top \Lambda_n^{-1}\varphi / \mathbb{E}[\text{lev}], 1)\) Calibrated BLR uncertainty (per-leaf)
prefix_distance(x) \((\text{prefix\_max} - \text{matched\_prefix}) / \text{prefix\_max}\) Routing-failure rate (per-graph)

The composite combines these into a single bounded [0, 1] score. Each signal catches a different OOD mode:

  • Density catches inputs that fall in a known leaf but far from its training cluster.
  • Epistemic catches inputs where the BLR posterior says “I’m uncertain here.”
  • Prefix distance catches inputs that don’t fit the codebook structure at all — i.e. the descent fell off the tree because no child exists at that prefix byte.

Why deliberately omit one child

A symmetric setup (children in all 4 bins, varying training density) would only exercise the density and epistemic signals — not prefix_distance. By deliberately not adding a child for bin 3, queries in bin 3 descend with matched_prefix=0, and the prefix-distance contribution to the composite fires.

The demo asserts the composite works, not just individual signals. If it only varied density, you’d be left wondering whether the routing-distance contribution is real or just a docstring claim. The asymmetric child set proves it’s real.

Capability 6 — Adaptive structural triggers

Demo: Two-sibling shared-signature topology (tests/test_abng_adaptive_triggers_cjcl.rs + adaptive_source.rs)

What this demo proves

  2. audit_log_grew — the structural mutation is recorded as an event in the chain.
  3. verify_chain == Ok — the chain integrity holds after structural mutation.
  4. chain_head matches the Rust-side decide_step_canary BYTE-FOR-BYTE — this is the most striking property in the entire demo set.

That last point deserves emphasis. The Rust-side decide_step_canary_tests.rs locks chain_head = d064fb08…7807. The CJC-Lang demo, running through both AST eval AND MIR exec, produces the same hex. The same workload, expressed three different ways (direct Rust API, CJC-Lang AST, CJC-Lang MIR), produces bit-identical SHA-256 output through 23+ events.

The math behind the Merge trigger

decide_step evaluates each node against six trigger conditions in order: Compress → Merge → Split → Prune → Grow → Freeze. For the Merge trigger specifically, ABNG checks two conditions on a pair of siblings (i, j):

  1. Signature similarity: Hamming distance of the 8-byte truncated SHA-256 signatures < tau_merge. The signature is computed from a 4-channel Welford profile (prediction, uncertainty, calibration, routing). Similar profiles → similar signatures → low Hamming distance.

  2. Posterior similarity (KL divergence): \[ \mathrm{KL}\bigl(\mathcal{N}(\mu_i, \Sigma_i)\,\|\,\mathcal{N}(\mu_j, \Sigma_j)\bigr) < \text{kl\_merge} \] where (μᵢ, Σᵢ) are the BLR posterior parameters at node i. This is the standard multivariate Gaussian KL.

Both conditions must hold for Merge to fire. The Merge action then folds j’s observations into i via Chan/Golub/LeVeque parallel Welford merge — preserving the post-merge stats as if they’d been observed in series.

Why the two-sibling shared-signature topology

  • Mirrors the Rust-side decide_step_canary exactly — root + 2 children at keys 1 and 2, three observations at root (0.10, 0.25, 0.40), three decide_step calls. This lets the CJC-Lang demo serve as a direct cross-pipeline determinism gate.
  • Merge is the most reliably-triggered action. With identical training distributions on the two siblings, their signatures align trivially (Hamming distance = 0). Decision-policy thresholds tau_merge=0 and kl_merge=0.1 are satisfied immediately — KL between fresh-prior posteriors is exactly zero.
  • Small graph keeps the test fast — 4 nodes, 3 observations, 3 decide_step calls. Runs in milliseconds even through the AST interpreter.

Capability 7 — Calibration / ECE

Demo: Hand-constructed well-calibrated vs miscalibrated outcomes (tests/test_abng_calibration_cjcl.rs + calibration_source.rs)

What this demo proves

Two graphs receive 30 calibration observations each. Same predicted probabilities (p ∈ {0.1, 0.5, 0.9}, 10 each), different outcome distributions:

Graph Outcome convention Expected ECE
A: well-calibrated predicted=0.1 → 1 of 10 positive; predicted=0.9 → 9 of 10 positive < 0.05
B: miscalibrated predicted=0.1 → 9 of 10 positive (FLIPPED); predicted=0.9 → 1 of 10 positive > 0.5

The headline assertion ece_a < ece_b separates the two cleanly. Plus the specific bounds: ece_a < 0.05 (well-calibrated achieves near-zero ECE) and ece_b > 0.5 (flipped outcomes produce huge ECE).

The math

ABNG uses standard 15-bin reliability-binning ECE (Naeini, Cooper, Hauskrecht 2015):

\[ \text{ECE} = \sum_{b=1}^{15} \frac{|B_b|}{N} \,\bigl|\,\text{accuracy}(B_b) - \text{confidence}(B_b)\,\bigr| \]

where:

  • B_b is the set of predictions falling in bin b (15 equal-width bins on [0, 1])
  • accuracy(B_b) = fraction of correct predictions in bin b
  • confidence(B_b) = mean predicted probability in bin b
  • |B_b| / N = bin’s empirical weight

A perfectly calibrated model has accuracy(B_b) = confidence(B_b) for every populated bin → ECE = 0. The miscalibrated graph with flipped outcomes pushes accuracy ≈ 1 - confidence in each bin → ECE saturates near 0.8.

Why “flip outcomes” as the miscalibration model

  • Maximum-ECE case. Predicting 90% confidence and being right 10% of the time is the worst possible outcome — confidence is anti-correlated with truth. This makes the test gate ece_b > 0.5 reliable; small numerical jitter doesn’t push the result close to the threshold.
  • Same prediction distribution → isolates the calibration signal. Both graphs see the same (p, y) pairs modulo y. Any difference in ECE is attributable to outcomes, not to predictions or to bin populations.
  • Three reliability bins (0.1, 0.5, 0.9) span low/mid/high probability range without requiring 100+ observations to populate every bin.

Capability 8 — Distribution-drift detection

Demo: Mean shift from 0 to 0.8 on a 4-D density tracker (tests/test_abng_drift_detection_cjcl.rs + drift_source.rs)

What this demo proves

A three-phase workload: train baseline, freeze, observe under same distribution, observe under shifted distribution. Three assertions:

  1. drift_grew — drift_score(post-shift) > drift_score(same-dist).
  2. meaningful_shift — drift_score(post-shift) > 2 × drift_score(same-dist). Catches a regression where drift detection becomes noisy.
  3. drift_nonneg — both scores are non-negative (no NaN, no negative noise).

The math

ABNG’s drift detection is built on a streaming diagonal-Gaussian density tracker. For features φ ∈ ℝᵈ, the tracker maintains:

  • Running mean: μ̂ updated via Welford
  • Running diagonal covariance: Σ̂_diag updated via Welford on (φᵢ - μ̂)²

freeze_drift_baseline(node) snapshots (μ̂_baseline, Σ̂_diag_baseline) at a reference point (typically end of pre-deployment training). After freezing, drift_score measures the divergence between the current tracker and the frozen baseline:

\[ \text{drift\_score} = \bigl\|(\hat{\mu}_{\text{cur}} - \hat{\mu}_{\text{baseline}})\bigr\|^2_{\text{Mahal}} + \alpha \cdot \bigl\|\Sigma_{\text{cur,diag}} / \Sigma_{\text{baseline,diag}} - \mathbf{1}\bigr\|^2 \]

— a weighted combination of mean shift (under baseline covariance) and covariance ratio deviation. Pure mean shift (this demo’s case) gives a score dominated by the first term.

Why a pure mean shift

  • Translation, not rescaling isolates the mean-shift contribution to drift_score. A combined “scale and rotate” shift would test more, but the basic mean-shift case is the simplest sufficient demonstration.
  • 0 to 0.8 is large enough to be unambiguous. Both distributions have ~0.01 jitter; shifting the center by 0.8 is 80x the jitter — the drift signal saturates well above the floor.
  • Same density-tracker geometry — same dimension count (4-D features), same covariance scale, same batch sizes (16 each post-baseline). Only the location changes.

The semantics

abng_node_maturity(g, node) returns a 4-element f64 tensor:

Index Field Type Definition
0 samples_seen non-negative count Total observations applied to this node
1 calibration_stable binary 0/1 True if 3-window ECE history’s \|Δ\| < ece_stability_max_delta
2 uncertainty_stable binary 0/1 True if 3-window σ history shows ratio < sigma_stability_ratio

The flags advance through decide_step calls plus the underlying signature_stable_calls counter that increments on consecutive matching signatures.

Why a stable observation stream

The maturity flags advance under two conditions:

  • signature_stable_calls increments when consecutive decide_step calls observe an unchanged signature

By choosing identical observations (abng_observe(g, n, 0.1) repeatedly), the signature stays exactly identical across decide_step calls — the stability counter climbs deterministically. With 10 observations + 5 decide_step calls, at least the stability flags should flip.

Realistic data (varying observations) would also work but with lower predictability — the signature might wobble enough that signature_stable_calls resets occasionally. The stable-stream choice makes the demo deterministic.

Capability 11 — End-to-end determinism

Demo: Universal property — every CJC-Lang demo asserts it.

What every demo proves

Every CJC-Lang demo runs through both cjc-eval (AST tree-walk) AND cjc-mir-exec (MIR register-machine), then asserts byte-equal printed output. Across the 13 ABNG demo files, this constitutes ~95 byte-equal parity assertions over a vocabulary of ~30 abng_* builtins.

The strongest single instance: the adaptive triggers demo’s chain_head matches the Rust-side decide_step_canary byte-for-byte (d064fb08…7807). Same workload, three pipelines (direct Rust API + AST eval + MIR exec), bit-identical SHA-256 over 23+ audit events.

The architectural choices that make this work

Three load-bearing decisions enforced across the workspace:

  1. Kahan compensated summation for all FP reductions (KahanAccumulatorF64). Naive Σxᵢ accumulates rounding error in O(n·ε); Kahan reduces this to O(ε). More importantly, Kahan is order-deterministic for fixed input order — no SIMD-introduced reordering, no FMA fusion. This is the single biggest source of “approximately deterministic” failure in standard ML stacks.

  2. BTreeMap/BTreeSet only, never HashMap. Hash maps with random hashing have iteration order that depends on the per-process seed — different runs can iterate in different orders, and any code that depends on iteration order (e.g., reduction loops) becomes non-deterministic. BTreeMap iterates in sorted key order, every time.

  3. SplitMix64 with explicit seed threading. No thread-local randomness, no time-based seeds, no platform-specific entropy. Every random draw is reproducible from (initial_seed, operation_history).

These three choices aren’t aspirational — they’re enforced by the parity gates. If anyone introduces FMA in cjc-runtime/linalg.rs tomorrow, every parity test fails simultaneously with byte-divergent output.

What this means for ML practitioners

ML reproducibility is famously hard. A typical “deterministic” PyTorch model can produce different bytes across runs due to:

  • CUDA atomic ops (non-deterministic reduction order on GPU)
  • cuDNN algorithm selection (can pick different kernels per run)
  • Multi-threaded CPU reductions (thread-scheduling non-determinism)
  • FMA fusion (compiler-dependent instruction selection)
  • Hash-randomization in Python’s dict

Each of these has been a real production-incident root cause. ABNG’s parity gates prove the absence of these bugs every test run.

What’s next — and why interpreter / compiler work matters

ABNG is genuinely useful for a specific niche: regulated ML training where you need Bayesian uncertainty + cryptographically-verifiable lineage on tabular data. The demos prove the niche capabilities; they don’t prove ABNG should replace PyTorch.

Three things stand between current ABNG and production readiness:

Cross-platform determinism CI

The locked canary hashes are Windows-only today. They should hold across platforms (the architecture is designed for it), but this is unverified. Adding GitHub Actions CI that runs the full test suite on Linux + macOS + Windows and compares the locked canaries is a one-day project that pays off forever — every future commit either preserves cross-platform determinism or fails CI loudly.

Performance benchmarks

The “linear scaling” claim for ABNG vs classical GP is correct asymptotically but not yet benchmarked at realistic dataset sizes. Without n ∈ {10³, 10⁴, 10⁵} wall-clock numbers, this is a “promising primitive” not a “GP killer.”

Compiler / interpreter performance

This is the bigger long-term constraint. The CJC-Lang interpreter is currently 5–50× slower than direct Rust API access for ABNG demos. For research / experimentation that’s fine. For production ML training at scale, it’s a hard ceiling.

The chess RL benchmark hit this wall directly — the bottleneck for v2.5 was interpreter throughput, not ML algorithm design. Specific levers, in increasing order of effort:

  • MIR optimizer passes — current optimizer has CF + DCE; further passes (loop-invariant code motion, common subexpression elimination, function inlining) are weeks of work each.
  • Hot-path specialization in cjc-runtime — ABNG already shows the pattern (75 native builtins for what users actually do). Has architectural cost: every native kernel is a feature you can’t compose out of simpler primitives.
  • AOT compilation — cjcl compile foo.cjcl → foo.exe. LLVM or Cranelift backend. Months of work but the only path that closes the perf gap meaningfully.

Most likely sequencing: continue ABNG (Phase 0.6 work — smart-replay fast-forward, cross-platform CI, lineage case study) while landing incremental MIR optimizer + native-kernel work. AOT is a multi-month commitment that depends on whether CJC-Lang’s long-term identity is research vehicle or production language.
